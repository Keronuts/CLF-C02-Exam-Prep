3. Video: Data Transfer Charges (it_clawsp23_15_enus_03)

  • describe data transfer charges
[Video description begins] Topic title: Data Transfer Charges. Presented by: Michael Shannon. [Video description ends]

Data transfer charges are a critical aspect of architecting an AWS cloud initiative. It is vital for customers to understand data transfer charges that they may come across when running production workloads at AWS.

There is no charge for inbound data transfer across all services in all regions at Amazon Web Services. Data transfer from AWS to the Internet is charged per service, with rates specific to the initiating region. In other words, they vary from region to region across the globe, and they're specific to that region.

Data transfer within AWS can be from the customer's workload to other AWS services, or it could be between different components of the customer workload. When a customer workload accesses AWS services, they may invite data transfer charges.

If an Internet gateway is used to access the public endpoint of the AWS services in the same region, there are no data transfer charges. If a Network Address Translation, NAT gateway is used to access the same services, then there is a data processing charge per gigabyte (GB) for data that passes through the NAT gateway.

[Video description begins] A diagram of Data Transfer Charges is displayed. There are two AWS Regions, one on the left and one on the right. Under each AWS Region, there are multiple services such as Amazon S3, Amazon DynamoDB, Amazon Kinesis, and so on. In the left AWS Region, under services, there's a VPC which includes Internet gateway. And in the right AWS Region, the VPC under services includes Internet gateway and NAT gateway. [Video description ends]

On the left hand side, we see a VPC using an Internet gateway to connect to various services in the region, such as S3, DynamoDB, SNS, and SQS. There's No data transfer charge. However, if the customer goes through a VPC. However, if the customer goes through a NAT gateway that has been deployed in the VPC, there's No data transfer charge, but there is a Per-hour service charge and per-GB data processing charge on the NAT gateway.

Data transfer within the same Availability Zone is free. One way to achieve high availability for a workload is to deploy it in multiple Availability Zones.

If customer workloads access services in different regions, there is a charge for data transfer across regions. The charges depend upon the source and destination region, for example, the Northern Virginia region in the US and the Paris region in France, as seen in the next slide.

[Video description begins] A diagram of Data Transfer Charges is displayed. There are two AWS Regions: AWS Region 1 is on the left and AWS Region 2 is on the right. In AWS Region 1, the VPC section includes NAT gateway connected to Internet gateway. AWS Region 1 is connected to AWS Region 2 through Per-GB charge for inter-region data transfer. In AWS Region 2, there are various services such as Amazon S3, DynamoDB, Amazon Kinesis, and so on. [Video description ends]

Let’s say AWS Region 1 is the Northern Virginia region in the United States, and AWS Region 2 is the Paris region in France.

If you're going through a NAT gateway to services running in the Paris region, for example, DynamoDB, Kinesis or Amazon Glacier, there’s going to be a Per-hour service charge and per-GB data processing charge generated from the NAT gateway and a Per-GB charge for inter-region data transfer.

Again, it depends on the source and destination region as to what those charges are. You don't have to know those charges for the cloud practitioner exam. Just realize there is an inter-region data transfer charge.

5. Video: Reserved Instance Behavior and Flexibility (it_clawsp23_15_enus_05)

Upon completion of this video, you will be able to describe reserved instance behavior and flexibility.

  • describe reserved instance behavior and flexibility
[Video description begins] Topic title: Reserved Instance Behavior and Flexibility. Presented by: Michael Shannon. [Video description ends]

Reserved Instance Behavior and Flexibility options. Amazon EC2 Reserved Instances (RIs) offer a substantial discount (up to 72%) when compared to On-Demand pricing, which is the default. RIs can deliver a capacity reservation, offering extra confidence in the ability to launch the number of instances reserved when needed.

Customers also have the flexibility to change families, OS types, and tenancies while benefiting from RI pricing when they use Convertible RIs.

Standard RIs provide the most significant discount (up to 72% off 0n-Demand) and are best suited for regular usage. Convertible RIs provide a discount (up to 54% off of On-Demand) but you sacrifice some of your discount by getting the capability to change the attributes of the RI if the exchange leads to the creation of Reserved Instances of equal or greater value. Scheduled RIs are available to launch within the time window you reserve.

The scheduling option allows customers to match their capacity reservation to a predictable recurring schedule when you only need part of a day, a week, or even a month.

Let’s compare Standard and Convertible. The Terms are 1 yr and 3 yr. Your savings with Standard are about 40% for 1 yr, 60% for 3 yrs. With Convertible, you give up some of your discount for that flexibility. 1 yr about 31% and 3 yr your savings is 54% .

You can change your Availability Zone, your instance type (for Linux OS), and your networking type for both.

With Standard, you use the ModifyReservedInstances API or the management console. With Convertible, you use the ExchangeReservedInstances API and/or the console.

Also notice, with the Convertible instance types, these are the only ones where you can Change the instance family, the operating system, the tenancy, and your payment options and still get a Benefit from price reductions. For billing purposes, the consolidated billing feature of AWS Organizations treats all of the accounts in the organization as one account.

All accounts in the organization can receive the hourly cost benefit of Reserved Instances that are purchased by any other account. Customers can turn off Reserved Instance discount sharing on the Preferences page on the Billing and Cost Management console.

7. Video: Cost Allocation Tags (it_clawsp23_15_enus_07)

[Video description begins] Topic title: Cost Allocation Tags. Presented by: Michael Shannon. [Video description ends]

Let's talk about tags at AWS. A tag is a label that a customer or Amazon assigns to an AWS resource. Every tag is made up of a key and a value. In other words, it's a key value pair for each resource. For example, an EC2 instance or an S3 bucket. The tag key must be unique. However, a key can have one to many values. Customers can use tags to organize their resources.

A special type of tag is a cost allocation tag. Cost allocation tags are used to track AWS costs at a granular level. After a customer activates a cost allocation tag, AWS uses the tag to organize their resource costs on a cost allocation report.

This makes it a lot easier for customers to categorize and track their AWS costs. We’ll see an example of a cost allocation report here in a moment. AWS offers two types of cost allocation tags: AWS-generated tags and User-defined tags.

AWS (or AWS Marketplace ISV) defines, creates, and applies the AWS-generated tags for the customer. The customer defines, creates, and applies their user-defined tags. Customers must activate both types of tags separately before they can appear in Cost Explorer or on a cost allocation report.

After the tag is applied to AWS resources (such as an EC2 instance or an S3 bucket) and the customer activates the tags in the Billing and Cost Management console, AWS will then generate a cost allocation report as a (CSV) comma-separated value file, with the usage and costs grouped by the active tags. Applied tags can represent business categories like cost centers, application names, or owners to organize costs across multiple services.

The cost allocation report includes all AWS costs for each billing period. The report includes both tagged and untagged resources so that customers can easily organize their charges for AWS resources.

For instance, if a customer tags resources with an application name, they can track the total cost of a single application that runs on those resources.

Here's a sample dummy cost allocation report where we can see we have the Total Cost that could be in whatever currency you’re using. And then we can see a key value pair. The key being User with several different values Owner, in this example, DbAdmin or SysEng. Stack which we have Test or Prod, the Cost Center value, the unique number, and Application value, Widget2, Widget1, or Portal.

And again the customer can look at the cost allocation report that's generated by AWS. Or you can go look at the Cost Explorer service.

8. Video: AWS Organizations (it_clawsp23_15_enus_08)

Upon completion of this video, you will be able to describe AWS Organizations.

[Video description begins] Topic title: AWS Organizations. Presented by: Michael Shannon. [Video description ends]

AWS Organizations. AWS Organizations provide policy-based management for multiple to many AWS accounts, including: creating groups of accounts, automating account creation using the console and APIs, and applying and managing policies for account groups. Organizations empower customers to streamline costs and take advantage of quantity discounts with a single consolidated bill.

Customers can centrally manage Service Control Policies (SCPs) within an organization across multiple accounts without having to use custom scripts, custom security policies, or manual processes. An SCP allows administrators to place guardrails on highly privileged principals with “deny” statements in the JSON-managed security policies.

[Video description begins] A diagram of AWS Organizations is displayed. The Master account/Administrative root is at the top and is connected to Dev, Test, and Prod Organizational units. They are further connected to AWS accounts, and lastly those AWS accounts are connected to AWS resources. [Video description ends]

User of an account can enable organizations in the identity and access management area, thereby creating the Master account/Administrative root. Then they’ll create Organizational units based on roles, programs, projects, responsibilities geographically.

Basically, whatever criteria they want to use for the Organizational unit, which is a container that holds one or more to many accounts and having several to many accounts is exactly what AWS wants you to have, as opposed to 1 or 2 accounts representing a small attack surface or 1 or 2 single points of failure.

Here we see the Dev OU, the Test OU, and the Prod OU. One advantage is that the Master account gets a single bill at the end of the month, with the charges broken down by Organizational unit. This way, they can chargeback or show back against the budget of the Dev, Testing, or Prod Organizational unit.

Also, the administrator can apply Service control policies at the OU level, for example, to put guardrails on a highly privileged groups or users in the Test account, basically denying certain API calls even though it’s allowed from a managed policy or a custom policy.
