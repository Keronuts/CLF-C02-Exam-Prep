AWS Cloud Practitioner 2023: AWS Network Services

When customers choose to use the AWS network services, they are leveraging some of the fastest and most cutting-edge datacenter and storage area networks (SANs) available. Many customers will begin the journey here since every account gets a default virtual network called a Virtual Private Cloud (VPC). In this course, you will begin by exploring the components of a VPC. You will then look at how to use VPC addressing, security, and peering and compare VPC endpoints with endpoint services such as PrivateLink. Next, you will learn about Amazon Route 53 and edge network services. Finally, you will discover connectivity options available for AWS, such as AWS VPN and Direct Connect. This course is one of a collection that prepares learners for the AWS Certified Cloud Practitioner (CLF-C02) exam.

Okay, in this lesson, we’re going to build on the concepts that we learned about in the previous two demonstration lessons and look at some AWS security.

Most of us are aware of the security triad: confidentiality, integrity, and availability (CIA). But AWS has a triad and their opinion is if you can gain mastery in all three of these areas, it will help you to have a more secure environment regardless of the services you decide to use and whether it be IaaS or PaaS or SaaS, moving forward in the AWS cloud.

The three elements are Identity and Access Management (IAM), Infrastructure security, specifically securing the VPC, and Key Management Service (the KMS).

There's three main elements to securing the VPC. Security of the VPC begins with the design of various subnets. Are the subnets public, private, or VPN only? Then the next step is to configure the different firewalls at layer 3 and layer 4 for internal traffic flow.

Then, the external connectivity options are considered next. Do you use a Internet gateway? Do you use the VPG, virtual private gateway for your Site-to-Site VPN? And other options like a secure Direct Connect connection?

Security begins with the design of your subnet, and the subnet depends upon entries in the route or the route table. [Video description begins] A loop diagram of VPC Security is displayed. The VPC section is connected to IGW and VPG, and they are both interconnected to Router. The Router part is divided into two sections: NACL subnet 1 and NACL subnet 2, before that there are two Route tables connected between the two subnets and Router. At the bottom, each subnet is further connected to Security groups and Instances. [Video description ends]

So for example, if you want subnet 1 to be a public subnet, then you’re going to put an entry to the internet gateway IGW into the route table. By default, the route table has a default entry saying this is your local traffic. For example, in this diagram it’s 172.31.0.0/16. The subnets have been broken up with /20 CIDR subnets. If you want a subnet to be VPN only, for example, subnet 2, you would put a VPG entry into the route table. A subnet cannot be public and VPN only.

In addition, if you have no IGW entry into the route table, nor do you have a VPG entry in the route table, it is a private subnet or a sandbox. The next step is Network ACLs.

Network ACLs allow stateless traffic filtering to all inbound or outbound traffic on a VPC subnet. It applies to all instances and components and appliances in the associated subnet. Typically, the NACL contains ordered rules to permit or deny traffic. The rules are processed with a numbered order, typically beginning with 100 and then incrementing by fives or tens.

The NACL is agnostic of TCP connections, UDP flows, or ICMP flows. It simply looks at the datagram or packet one at a time, otherwise referred to as atomically, and makes a decision based on the headers. If there is a match, it permits or denies. They are stateless or (static), and that the return traffic must also be explicitly allowed in the inbound NACL.

These firewalls work together as a defense in depth mechanism with security groups, and they can permit or deny traffic before it reaches the interfaces of the elements inside the subnet. [Video description begins] The same loop diagram is displayed again. [Video description ends]

Notice in the diagram, not only does the subnet have one and only one route table, but a subnet can only have one NACL. Can two subnets or more share the same route table? Yes, they can, in the same way that two or more subnets can share the same NACL. However, they have only one NACL with an inbound rule set and an outbound rule set.

Keep in mind that by default, in the default VPC, the NACL is allowing all traffic inbound and outbound. In other words, it's not activated. It's an additional defense in depth mechanism. The security group, however, is activated.

A Security group (SG) firewall applies to each individual EC2 instance in a subnet. They actually function at the hypervisor level. They’re attached to the virtual elastic network interface, by default, eth0. Security groups or SGS are layer 3/4 stateful virtual “Allow Only” firewalls.

In other words, they're allow lists and they have no explicit deny rules like the NACLs. All EC2 instances are launched with the default SG unless otherwise designated. An unchanged default SG will allow communication between all resources within the security group AND all outbound traffic.

However, all other traffic is implicitly denied. The return traffic is automatically allowed (with AWS Shield Standard Inspection). All rules in all applied SGs are evaluated before a decision is made. In other words, these are not numbered lists. Everything in the group is evaluated.

There are limits or (quotas) on the number of SGs per VPC, on the number of rules you can add to a security group, and on the number of SGs that you can use with an elastic network interface.

Earlier we mentioned that a subnet can only have one NACL. However, an interface can have two or more security groups. [Video description begins] One more time, the same loop diagram is displayed. [Video description ends] Notice in this diagram, subnet 1 has three instances of Windows, Linux, MacOS, or a virtual appliance. Each one can have its own Security group. Notice in subnet 2, we have two instances, let’s say, Linux. And they're sharing the default Security group.

A Web Application Firewall or WAF, helps you protect against common web exploits and bots that can affect availability, compromise your security, or otherwise consume excessive resources.

A WAF is a Layer 7 web access control list, or web ACL, that uses security rules to control traffic and block attack patterns such as SQL injection, request forgeries, or cross-site scripting (XSS) variants. You can apply a WAF to protect your web applications from exploits on an Amazon CloudFront distribution, on an Application Elastic Load Balancer, on an Amazon API Gateway, or you can apply it, for example, in a hybrid cloud solution on the AWS AppSync instance.

To use a WAF, you create a policy or more often than not, you deploy managed rules that are maintained by AWS or AWS Marketplace sellers. These rules will block and filter. They protect against vulnerabilities and exploits such as SQLi or cross-site scripting attacks, and they filter out unwanted traffic by IP address, geographic location, or by defining specific patterns in regular expressions.

Then, you can use Amazon CloudWatch to receive incoming traffic metrics for visibility and monitoring. Or you can use Amazon Kinesis Data Firehose to request details and tune rules based on metrics and log data. Or, you can rely upon the AWS Marketplace sellers to do all of this for you.

VPC peering connections can be between your own VPCs or with a VPC in another AWS account. Intra-region or inter-region peering connections are allowed. In the diagram, to create a peering connection between VPC A and VPC B, there has to be a requester and an acceptor. If an individual is an administrator and has the permissions if both VPCs are in the same account, somebody with permissions can be the requester and the acceptor.

If the other VPC is in a different account and that same person has permissions, they can also be the acceptor and the requester. It’s a best practice and recommended. If the two different VPCs in the peering connection use different IP prefixes, they can overlap. However, there has to be some additional configuration done on route tables in both VPCs.

Also, realize that VPC peering is not transitive. So for example, if there were a third VPC, VPC X and VPC A had a peering connection with X and VPC B had a peering connection with X. It’s non-transitive. Instances in VPC A could not go through VPC X to instances in VPC B and vice versa. They’d have to have an explicit peering connection between them. Amazon Route or Route 53. Route 53 is a highly available and scalable cloud domain name system (DNS) web service. It's designed to offer a very dependable and cost-effective way to route end users to Internet applications by translating human-readable domain names to IPv4 and/or IPv6 addresses.

Route 53 effectively connects user requests to AWS services like EC2 instances, Elastic Load Balancing load balancers, typically application load balancers, and Simple Storage Service (S3) buckets, which can be accessed through URLs or APIs. It can also be used to route users to infrastructure elements outside of AWS. For example, headquarters, regional branch offices, or co-location environments. Let's see how it works.

[Video description begins] A flowchart of Amazon Route 53 is displayed. At the top, there are two sections: AWS Management Console and Amazon Route 53 SDK/API. At the bottom, there's only one section, AWS CloudWatch. [Video description ends] An End User sends a request. Route 53 has a DNS resolver which resolves domain name queries on behalf of the end users. Amazon Route 53 has an Authoritative DNS Service, so it will return IP addresses for DNS records queried by the DNS resolver.

Eventually, the request gets to the endpoint instances. For example, the EC2 instance running Windows, Linux, or macOS, or maybe Red Hat Enterprise running a container. However, there are other services in play. For example, you can use the AWS Management Console for hosted zone creation, traffic policies, and the propagation of DNS records to the Route 53 global network of authoritative DNS servers.

You can also use the SDK (Software Development Kit)/API with Route 53. Route 53 also performs health checks to monitor the health of your endpoints for high availability. And of course, you can leverage AWS CloudWatch, delivering monitoring metrics for Route 53 and generating alarms as well as Lambda functions if necessary.

In this lesson, we’re going to look at Edge Network Services at AWS, starting with Amazon CloudFront. CloudFront is a content delivery or distribution network (CDN) service designed for high performance, security, and high suitability in DevOps environments, for example, web applications and containers.

It securely delivers data, videos, applications, and application programing interfaces (APIs) to customers globally with low latency and high transfer speeds within a developer-friendly environment. It’s integrated with the AWS cloud - both physical locations that are directly connected to the AWS global edge locations and various service endpoints in metropolitan areas all over the world.

[Video description begins] A vertical flowchart of Amazon CloudFront is displayed. It's divided into 5 sections, End user traffic, AWS Shield and AWS WAF, Regional edge caches, Origin shield, and Application content origin. And two sub-sections under Regional edge caches are labeled: Amazon CloudFront and Lambda@Edge. [Video description ends]

When using CloudFront, the End User requests will go to the nearest AWS edge location, where CloudFront caches, typically in Redis ElastiCache, and serves static and dynamic content. It also provides < 1 millisecond serverless edge functions. At the edge locations, you have AWS Shield, standard and advanced, for Anti-DDoS and DDoS protection, and the AWS Web Application Firewall to protect against OWASP top ten attacks against web applications.

AWS also has regional edge caches in cities all over the world, where CloudFront leverages globally dispersed mid-tier caching for longer cache retention, and Lambda@Edge to support serverless edge functions. Then, Origin shield improves network performance and cache-hit ratio to the origin content, which could be an S3 buckets, EC2 instances, you could also go through the API Gateway, Elastic Load Balancers, AWS Media Services, or even data stored on-premises or in other providers.

Another Edge Network Service is Global Accelerator. This is a robust networking service that enables customers to expand the availability, performance, and security of their public applications. It offers two global static public IPs that represent a fixed entry point to application endpoints such as the Elastic Application Load Balancers, Network Load Balancers, Amazon Elastic Compute Cloud (EC2) instances, and even elastic IPs (eIPs).

Some use cases for Global Accelerator would be using traffic dials as a global traffic manager to route traffic to the nearest region or attain fast failover across multiple regions. You can accelerate API workloads by up to 60%, leveraging TCP termination at the Global Accelerator edge. Simplify allowlisting in enterprise firewalling and IoT scenarios using global static IPs.

And use custom routing techniques to deterministically route traffic to a fleet of EC2 instances for low-latency gaming and media services.
