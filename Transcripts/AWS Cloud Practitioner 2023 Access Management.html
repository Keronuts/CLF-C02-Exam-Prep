AWS Cloud Practitioner 2023: Access Management

A large aspect of the daily responsibilities of cloud security involves monitoring, controlling, and optimizing identity and access management (IAM) of cloud resources and services. In fact, IAM is at the top of the AWS security triad pyramid; the other two being infrastructure security and key management. In this course, you will explore the key concepts of IAM at AWS, beginning with the least privilege principle and root account protection. Then you will discover how to work with AWS IAM, AWS IAM Identity Center. Finally, you will compare multi-factor authentication options and define cross-account roles. This course is one of a collection that prepares learners for the AWS Certified Cloud Practitioner (CLF-C02) exam.

2. Video: Least Privilege Principle (it_clawsp23_06_enus_02)

After completing this video, you will be able to define the least privilege principle.

  • define the least privilege principle
[Video description begins] Topic title: Least Privilege Principle. Presented by: Michael Shannon. [Video description ends]

Well, we can't talk about Identity and Access Management at Amazon Web Services without first exploring the least privilege principle. Least privilege is the principle that a security architecture should be utilize so that each subject, otherwise known as a principal, is granted the minimum system resources and authorizations that the entity needs to conduct its activities, and nothing more. When related to data and information, this is often called the "need to know" principle. At Amazon Web Services, this is often applied with the role-based access control (RBAC) model unless federated single sign-on (SSO) is being used. When setting permissions with Identity and Access Management policies, grant only the permissions necessary to perform the task.

In other words, only allow the API calls through the AWS Customer Metastructure to the resources in AWS data centers needed to achieve a specific goal. This is accomplished by defining the actions that can be taken on specific resource objects under certain conditions. Administrators often begin with broader permissions, then subsequently explore the permissions that are needed for a particular workload or use case. This least privilege principle is also referred to as the least privilege strategy or LPS. It's common for users to have broader or elevated permissions using their own on-premises identity provider but have lesser privileges with resources in the cloud. A critical element of a Zero Trust initiative (ZT) is to adopt a least privilege strategy (LPS) that enforces access control to eliminate the human temptation to access restricted resources.

3. Video: Root Account Protection (it_clawsp23_06_enus_03)

Upon completion of this video, you will be able to describe distinctives of the root account and methods for protection.

  • describe distinctives of the root account and methods for protection
[Video description begins] Topic title: Root Account Protection. Presented by: Michael Shannon. [Video description ends]

In this lesson, we're going to talk about protecting the root account at AWS. The root account is a standalone single sign-on unrestricted access entity. Notice on the Sign in page, the interactive user logging on to the console has the choice of signing in as the Root user or as a user created in the Identity and Access Management service of AWS. When the customer first creates an AWS account, they begin with a single sign-in entity that has complete access to all AWS services and resources in the account. This identity is called the AWS account root user and is accessed by signing in with the email address and password used to create the account.

Hopefully, the password is long and strong and well-protected. Keep in mind that the password policy that may be configured in the IAM service does not apply to the root account. The root account is subject to the AWS default password policy, which can change over time. Customers are strongly discouraged from using the root user for everyday tasks (even the administrative ones). Root user credentials should only be used to perform a few account and service management tasks. Some of the distinctive activities that can only be performed by the root account user would include changing the support plan, for example, from a business plan to an enterprise on-ramp plan. Modifying payment options and billing, adding a credit card, removing a debit card, or going to direct billing. This includes creating an organization, closing an AWS account.

Signing up for AWS GovCloud in the United States, transferring a Route 53 domain to another account. This can only be done by the root account user. And creating an AWS-created X.509 signing certificate. The root account should not be accessed programmatically. In other words, the root account user shouldn't get an access key ID and an access key and then manage the account through a command line interface. If they have been created, then delete them immediately. And always use a physical or software-based multi-factor token access on the root account. We will look at multi-factor authentication options at Amazon Web Services and a lesson coming up later on in this course.

8. Video: Cross-account Roles (it_clawsp23_06_enus_08)

After completing this video, you will be able to define cross-account roles.

  • define cross-account roles
[Video description begins] Topic title: Cross-account Roles. Presented by: Michael Shannon. [Video description ends]

In AWS Identity and Access Management, a role is just a logical object that has permissions assigned to it. Now, we know that our managed policies are just JSON documents that serve as allow lists of what API calls can be made. In other words, what can an entity do through the metastructure in the controller in the data center of Amazon Web Services? Here we see that a policy, a JSON document with a set of API allows can be applied to a group, for example, the admins group in account A, and that's very common. This is scalable and more centralized. We create users Amber and Max and we just place them in the group and they assume the policy. Now, a policy can also be assigned directly to a user.

For example, we can assign the PowerUserAccess policy to Gina in account B and the AdministratorAccessPolicy to a user Cody in account B. We can also assign the PowerUserAccess policy to a single user Lucas in account A. However, that's only done on a limited basis, and really for a small group or a small account. What we're talking about here though is that a managed policy can actually be assigned to a role. A role is a logical object, it's just a container, and you can assign a policy to that role. In fact, you can assign up to 10 policies to a role or a group, or a user and it would be the aggregate or the accumulation of all of the API calls that are allowed in the JSON documents.

With roles, we assign the policy to the role and then some entity will assume that role and they'll be able to access with a token the resource for a certain period of time and only be able to do the API calls that are in the managed policy. On the exam, you'll be tested on cross-account roles. So, let's say, we have two accounts. We have the development account on the right-hand side and the production account on the left-hand side. In the development account, we have two groups, the Developers group and the Testers group. And in the production account, we deploy our live applications. Let's say in the development account, we have a user named Ted. We want to give Ted access to a S3 bucket in the production account called myapp.

What we're not going to do is we're not going to create another user called Ted in the production account, which happens to be the trusting account. It's where the things are that's the one that's going to be trusting Ted. So, the way a cross-account role works is this. You create a role in the trusting account or the production account called myapp, and it's just simply a role or a container, and then you assign permissions to that role, manage permissions. In other words, what API calls can somebody who assumes that role in the production account? Once that's done, an administrator in the development account will actually assign that role to Ted, and then Ted will temporarily assume that role, get a token, and then access that S3 bucket for a certain period of time, but only making the API calls against that S3 bucket in the production account that he's allowed to do based on the manage permissions assigned to the role in the trusting account that he's assumed. That's how cross-account roles can work.
