2. Video: Block Storage vs. Object Storage (it_clawsp23_12_enus_02)

[Video description begins] Topic title: Block Storage vs. Object Storage. Presented by: Michael Shannon. [Video description ends]

There are three types of cloud storage at Amazon Web Services, object, file, and block, each having specific use cases and storage needs. Block storage or volume storage is a classic technology that manages data storage and storage devices. For example, on your traditional PC, workstation, laptop, or RAID array. It takes any data, like a file or a database entry, and distributes it into equal size blocks (512-bit is legacy, 4K is more common). The block storage system stores the data block on supporting physical storage, for example, hard disk drives and solid-state drives, in a way that's optimized for fast access and retrieval, for example, from a RAID array, Redundant Array of independent disks. Developers prefer block storage for applications that require efficient, fast, and reliable data access.

Block storage offers a more direct pipeline to the data. By comparison, file storage has an extra layer consisting of a file system (NFS, SMB, or CIFS) to process before accessing the data. Let's look at some common use cases for block storage. Storage area networking (SANs), containerized applications, transactional workloads, analytics and data warehousing, and virtual machines. In this diagram, we'll see all three types at Amazon Web Services. For example, let's say, instance B is an instance of Red Hat Linux. When it gets instantiated or spun up, it'll have a system volume, an Elastic Block Store volume, and usually a separate data volume, also Elastic Block Store. If for some reason you stop running the Red Hat instance or you terminate it, you don't lose the data in the Elastic Block Store. It's part of your account.

Also, usually by default, when you create an Elastic Block Store volume, AWS or the database administrator will create a snapshot in an S3 bucket. That's object storage. What if instance B doesn't want to store its subsequent files on an Elastic Block Store volume? That's OK. It can have an endpoint connection to Amazon EFS, the Elastic File System, which is an example of file storage. There's also an instance store that's ephemeral at AWS. For example, let's say instance A is a high performance compute instance doing data analytics, floating point calculations, and creating many temporary files. Often it will use an array of solid-state drives, that's an ephemeral instance store that's directly attached to the instance. If you stop running that high performance compute instance, however, you lose the data on the instance store because it's ephemeral.

Let's look closer at object storage. Object storage is a technology that stores and controls data in an unstructured format called objects. Modern enterprises generate and analyze large amounts of unstructured data such as photographs, graphics, audio files, video files, bulk email, web pages, and data generated from sensors, for example, log data. Cloud object storage systems such as Amazon Simple Storage Service (S3) and S3 Glacier archives distribute this data across multiple physical devices but let users access the content proficiently from a single, virtual storage pool.

Object storage solutions are excellent for building cloud-native applications that require scale and flexibility and can also be used to import existing data stores for analytics, backup and restore, or archival solutions. Metadata is critical to object storage technology, and objects are kept in a bucket instead of files or folders. Object storage combines the chunks of data that make up a file, adds all the user-created metadata to that file, and then assigns a custom identifier. This scenario creates a flat structure instead of a hierarchical or tiered storage structure. Users can get and analyze any object in the bucket, no matter the file type, based on its function and attributes.

You set it up by defining settings and initiate continuous data replication. You can launch instances for non-disruptive tests. You can maintain operational readiness with monitoring and periodic drills and tests. And you can test your failover launch recovery instances on AWS and initiate failback replication and return to the primary site. When you use AWS Elastic Disaster Recovery, your data is replicated to a staging area subnet in your AWS account, in the AWS region that you choose.

It's called CloudEndure Disaster Recovery and on the exam, remember that they're going to recommend CloudEndure Disaster Recovery only if you need the following capabilities.
