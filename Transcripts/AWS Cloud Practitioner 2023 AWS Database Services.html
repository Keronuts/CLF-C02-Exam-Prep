AWS Cloud Practitioner 2023: AWS Database Services

Database is one of the core foundational services of AWS. A database is an electronically stored, systematic collection of data including words, numbers, images, videos, and files. In computer systems, the word database can also refer to any database management system (DBMS), to the database system, or to an application associated with the database. In this course, you will begin by exploring the use of relational, NoSQL, and memory-based databases in AWS. Next you will compare database migration tools including AWS Database Migration Service (AWS DMS) and AWS Schema Conversion Tool (AWS SCT). You will also compare Amazon Elastic Compute Cloud (EC2) hosted databases to AWS managed databases. Finally, you will explore the use of Amazon Aurora and Amazon Neptune. This course is one of a collection that prepares learners for the AWS Certified Cloud Practitioner (CLF-C02) exam.

2. Video: Relational Databases (it_clawsp23_10_enus_02)

After completing this video, you will be able to provide an overview of relational databases at AWS.

  • provide an overview of relational databases at AWS
[Video description begins] Topic title: Relational Databases. Presented by: Michael J. Shannon. [Video description ends] A relational database is a collection of structured data items with pre-defined relationships between them. These database items are ordered as a set of tables with associated columns and rows. Tables are used to hold information about the objects to be represented in the database. Each column in a table holds a specific format of data, and a field stores the actual value of an attribute.

The rows in the table embody a collection of related values or attributes of one object or entity. Each row in a table could and should be marked with a unique identifier, often called a primary key, and rows among multiple tables can be made related using foreign keys. Subsequently, this data can be accessed in many ways without rearranging the database tables themselves, and this rearranged data can be viewed as an abstraction of the underlying data.

Structured Query Language, SQL or Sequel is the main interface used to communicate with relational databases. Data integrity is the general fullness, accuracy, and consistency of data. Relational databases use a set of constraints called keys to enforce data integrity in the database.

A database transaction is one or more all or nothing SQL statements that are performed as a series of operations that establish a single logical unit of work. To ensure data integrity, all database transactions must be ACID compliant: a atomic. In other words, singular, granular, like a single IP packet or datagram. Consistent, isolated, separate, and compartmentalized from other data, and durable, not subject to loss, corruption, or lack of integrity.

In the next lesson, we'll look at a different type of database called a document or NoSQL database.

[Video description begins] Topic title: NoSQL Databases. Presented by: Michael J. Shannon. [Video description ends] NoSQL databases are purpose-built for designated or customized data models. They have elastic schemas for constructing modern applications, web apps, mobile apps, container support, and more. They're commonly known for their simplicity of development functionality and scalable performance.

NoSQL databases can use a diversity of data models for accessing and controlling the data. These database types are optimized explicitly for applications that demand large data volume, like data lakes, low latency or fast speed, and support of flexible data models. This is accomplished by bypassing some of the data consistency restrictions of other databases, such as the relational type that we discussed in the aforementioned lesson.

NoSQL databases are an excellent solution for many modern applications like mobile, web, and gaming. They offer more flexible schemas that enable faster and more iterative database development. They're generally designed to scale out by using distributed clusters of hardware, instead of scaling up by adding expensive and more robust servers.

They're optimized for high performance for specific data models and access patterns that allow for higher performance than relational databases. And they offer highly functional APIs, Application Programming Interfaces, and data types that are purpose-built for their respective data models.

Amazon DynamoDB. Dynamo is a fast, malleable NoSQL database service for single-digit millisecond performance at any scale. It's a fully managed, serverless, key value NoSQL database that was designed by Amazon to run high performance applications. It delivers built-in security, nonstop backups, automated multi-Region replication, in-memory caching, and powerful data import and export tools.

As a customer of AWS, you can configure key features of Amazon DynamoDB. It has built in security, native backup and restore, flexible capacity modes, the ability to replicate in a multi-Region deployment, in-memory caching, data modeling tools, and more like a NoSQL workbench. Global tables that are multi-Region, native encryption at rest with AES-256. Point in time recovery and on demand capacity mode and support for PartiQL. Then you can export, analyze and stream the data by integrating with other AWS services. You can export table data to conduct analytics and extract insights, or you can monitor trends and logs for enhanced security by storing it in Amazon S3 buckets or leveraging AWS Glue Elastic Views, analyzing with Amazon Kinesis Data Streams, monitoring trends with AWS CloudTrail and getting visibility with Amazon CloudWatch.

4. Video: Memory-based Databases (it_clawsp23_10_enus_04)

After completing this video, you will be able to provide an overview of memory-based databases such as Redis ElastiCache.

  • provide an overview of memory-based databases such as Redis ElastiCache
[Video description begins] Topic title: Memory-based Databases. Presented by: Michael J. Shannon. [Video description ends] Memory-based databases. In-memory databases are purpose-built databases that typically depend on high-speed memory chip clusters for data storage, as opposed to databases that store data on disk or solid-state drives, SSDs. In-memory data storage is intended to allow for nominal response times by abolishing the need to access physical disks.

Since all of the data is stored and managed exclusively in main memory. Memory based databases do risk losing data if there is a process or server failure. This is often called ephemeral storage. In-memory databases can persist data on chips or even disks by storing each operation in a log or a snapshot. In-memory databases are best for caching and applications that need microsecond response times or have big spikes in traffic like gaming, leaderboards, session stores, and real-time analytics.

A popular in-memory database at Amazon is ElastiCache for Redis. ElastiCache is an extremely fast in-memory data store that provides sub-millisecond latency to enable internet scale, real-time applications. ElastiCache is constructed on open-source Redis and compatible with the Redis APIs. Self-managed Redis applications can function effortlessly with Redis ElastiCache without any code changes, and it joins the speed, ease, and flexibility of open-source Redis with manageability, security, and scalability from AWS.

ElastiCache for Redis is used for real-time apps and gaming, ride-hailing. For example Uber or Lyft, media streaming, dating and social media that need fast data access. ElastiCache for Redis provides blazing fast in-memory data store for use as a database, a cache memory broker and queue, as well as storing ephemeral data in-memory for sub-millisecond response. Other use cases are real time transactions, chat and chat bots, business intelligence and analytics, session store, gaming, leaderboards, and cache. For example, at AWS edge locations to support content distribution networking.
